# Sprint 7 — Check-list lancement pilote

Objectif :
- préparer un lancement pilote simple et réaliste
- vérifier les points les plus utiles avant un usage réel encadré
- éviter un process de déploiement lourd ou une documentation trop longue

## Format retenu

Usage conseillé :
- 1 organisation pilote clairement identifiée
- 1 référent principal côté client
- 2 à 5 utilisateurs pilotes maximum au démarrage
- un scénario d'usage court et concret

Lecture attendue :
- cocher `OK` si le point est prêt
- cocher `À revoir` si le point doit être corrigé avant lancement
- cocher `Non concerné` si le module ou le cas n'est pas utilisé

Règle simple avant ouverture :
- aucun point bloquant sur `accès`, `modules activés`, `scénario pilote`, `feedback`, `exports utiles`

## Check-list

### 1. Périmètre pilote

- [ ] `OK` / `À revoir` / `Non concerné` : l'objectif du pilote est formulé en une phrase simple
- [ ] `OK` / `À revoir` / `Non concerné` : le ou les modules testés sont clairement choisis
- [ ] `OK` / `À revoir` / `Non concerné` : les parcours à montrer ou à utiliser en réel sont limités et connus

### 2. Pré-requis produit

- [ ] `OK` / `À revoir` / `Non concerné` : l'organisation de test ou pilote est prête
- [ ] `OK` / `À revoir` / `Non concerné` : les modules nécessaires sont activés
- [ ] `OK` / `À revoir` / `Non concerné` : les données minimales utiles existent déjà
- [ ] `OK` / `À revoir` / `Non concerné` : les messages visibles et statuts clés restent compréhensibles

### 3. Comptes et personnes

- [ ] `OK` / `À revoir` / `Non concerné` : les comptes testeurs existent
- [ ] `OK` / `À revoir` / `Non concerné` : chaque personne sait sur quelle surface elle intervient
- [ ] `OK` / `À revoir` / `Non concerné` : un référent pilote côté client est identifié
- [ ] `OK` / `À revoir` / `Non concerné` : un référent Conforméo est identifié pour centraliser les retours

### 4. Scénario pilote

- [ ] `OK` / `À revoir` / `Non concerné` : un scénario de démonstration ou de prise en main court est défini
- [ ] `OK` / `À revoir` / `Non concerné` : au moins un cas d'usage réel est choisi
- [ ] `OK` / `À revoir` / `Non concerné` : le parcours ne demande pas de navigation complexe inutile

### 5. Contrôles fonctionnels avant lancement

- [ ] `OK` / `À revoir` / `Non concerné` : le cockpit donne une lecture utile du contexte
- [ ] `OK` / `À revoir` / `Non concerné` : les chantiers utiles sont relisibles sans ambiguïté
- [ ] `OK` / `À revoir` / `Non concerné` : les documents chantier utiles sont retrouvables
- [ ] `OK` / `À revoir` / `Non concerné` : les exports PDF nécessaires au pilote fonctionnent
- [ ] `OK` / `À revoir` / `Non concerné` : la coordination légère reste compréhensible si elle est utilisée
- [ ] `OK` / `À revoir` / `Non concerné` : la synchronisation visible reste rassurante si le mobile est concerné

### 6. Recueil de feedback

- [ ] `OK` / `À revoir` / `Non concerné` : le canal de retour pilote est défini
- [ ] `OK` / `À revoir` / `Non concerné` : le panneau desktop `Donner un retour` est montré si utile
- [ ] `OK` / `À revoir` / `Non concerné` : les retours attendus sont limités à l'utile (`bloquant`, `incompréhension`, `amélioration`, `retour positif`)

### 7. Cadrage de lancement

- [ ] `OK` / `À revoir` / `Non concerné` : la date ou fenêtre de démarrage du pilote est posée
- [ ] `OK` / `À revoir` / `Non concerné` : la durée d'observation initiale est connue
- [ ] `OK` / `À revoir` / `Non concerné` : le critère simple de réussite du pilote est défini

## Sortie attendue avant ouverture

Avant de lancer le pilote, il doit rester :
- zéro blocage sur le parcours principal
- un canal clair pour remonter les retours
- un périmètre court, assumé et compréhensible

À relever après le premier passage réel :
- les blocages réels
- les incompréhensions les plus fréquentes
- les parcours qui rassurent le plus

## Hors périmètre

Ce document ne couvre pas :
- un plan de déploiement complet
- une conduite du changement détaillée
- un outil de pilotage de lancement
- une documentation de formation exhaustive
